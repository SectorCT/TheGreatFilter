# Qlean PRD

## Hack TUES 2026

**Version:** 1.1  
**Product Type:** Desktop application  
**Platform:** Electron + Vite + React  
**Document Scope:** Product features and expected user experience (high-level only)

## 1. Product Vision
Qlean is a desktop application that helps users work with water quality data and generate custom filtration designs from selected measurements.  
The product focuses on a clear user flow:
- access the app securely
- collect or select water measurements
- trigger filter generation
- track generation progress
- review generated and previous filters

This document defines what users should be able to do and what the product should look like at a high level, without defining code-level or API-level contracts.

## 2. User Flow Overview
1. User opens the desktop app.
2. If not authenticated, user sees Login / Sign Up.
3. If authenticated, user lands on the Dashboard.
4. User reviews existing data (previous filters + water measurements).
5. User adds or selects a measurement source.
6. User starts a new filter generation from a chosen measurement.
7. User tracks filter status until generation completes.
8. User opens and reviews the generated filter details and visualization.

## 3. Core Application Areas

### 3.1 Authentication Entry
**Goal:** Provide secure access to personal data and generated filters.

**Expected experience:**
- On app launch, unauthenticated users see:
  - Login
  - Sign Up
- Authenticated users skip the auth screen and go directly to Dashboard.

### 3.2 Dashboard
**Goal:** Give users one central place to manage measurements and filters.

**Dashboard content:**
- Previous filters generated by the current user
- List of water measurements owned/selected by the user
- Entry points for adding a new water measurement
- "All Filters" area with status labels

**Filter list behavior:**
- Completed filters are visible in the list
- In-progress filters are also visible in the same list
- In-progress statuses are shown as:
  - `Pending`
  - `Generating`

### 3.3 Add Water Measurement
Users can add a measurement in multiple ways.

#### A) Manual Input
**Purpose:** Quick direct data entry by a user.

**Validation rules:**
- Required fields:
  - Temperature
  - pH
- All other measurement fields are optional.

#### B) Lab equipment via USB
**Purpose:** Import a full measurement from a connected hardware sensor setup.

**Expected experience:**
- User selects the USB lab equipment input method
- App reads and creates a complete measurement record
- Measurement appears in the dashboard list

#### C) Map-Based Selection (GemStat Dataset)
**Purpose:** Use historical/current data from known measurement centers.

**Expected experience:**
- User opens a map of available GemStat locations
- User clicks a location (measurement center)
- User can view:
  - historical water quality
  - current water quality
- User selects location + date to retrieve the latest available data for that context
- Selected dataset snapshot can be used as the measurement input for filter generation

#### D) Import From CSV
**Purpose:** Let users bring measurements from external sources and start filter generation without manual re-entry.

**Expected experience:**
- User selects a CSV import action
- App imports a measurement record from the CSV file
- The imported measurement becomes available in the dashboard measurements list
- The app displays import validation feedback (e.g., missing `Temperature` or `pH` if they are required for manual entry in this workflow)

## 4. Water Measurement Model (Product-Level)
Each measurement entry represents what is being measured, the parameter identity, and the unit.

At a high level, each measurement item includes:
- Parameter code
- Parameter name
- Unit
- Value

The app supports a broad catalog of water quality indicators (for example from GemStat), including chemical, biological, physical, and pollutant-related parameters.

## 5. Filter Generation Experience
**Goal:** Allow users to generate a new filter from a selected measurement.

**User-facing behavior:**
- User chooses a measurement
- User starts "Generate Filter"
- Generation may take significant time
- App confirms request acceptance
- App periodically refreshes filter status
- Once status is successful, filter details become available
- User can open and view filter information and a visualization view (defined as a product feature)

**Export behavior (high level):**
- After a filter is generated successfully, the app provides an option to export the generated filter as a CSV file
- The CSV export format is intentionally to be defined later
- Multiple visualization options may be added later, but CSV export should remain available as a stable output option

**Important scope note:**  
The detailed data contract for generation requests and responses remains intentionally undefined at this stage.

## 6. Visualization Feature (High-Level)
Visualization is part of the product experience and should allow users to inspect generated filter output in an intuitive way.

At this stage, only high-level behavior is defined:
- generated filters should be visually explorable
- visualization should be accessible from the filter details flow
- detailed rendering format is to be defined later
